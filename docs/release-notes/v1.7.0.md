# Toolport v1.7.0

**Security and reliability batch.**

The headline is a scope-resolution fix: a client scoped to a profile that has since been
deleted or renamed used to fall back to the active profile, silently widening its view to
that profile's servers. It now fails closed to an empty server set. An unscoped client
(empty reference) still follows the active profile as before.

Alongside it, a batch of reliability and quality-of-life work:

- **Per-server working directory** — pin a stdio server (filesystem/grep tools) to a
  project directory instead of inheriting Toolport's. `~` and `${VAR}` expand.
- **Live per-client profile scoping** — re-scope a client from the app and it applies on
  the next reload, no client restart. Plus diagnostics that flag a connected-but-
  unauthenticated server (enabled, 0 tools).
- **Incremental server grid** — one cold `npx`/`uvx` install no longer freezes the whole
  grid at "checking" for 30-60s; each row resolves as its own probe finishes.
- **Readable error rows** — a failed probe leads with a one-line headline instead of a
  Node stack trace and a multi-KB OAuth URL, with a Copy button for the full output.
- **Registry hardening** — a rolling backup journal (newest-5) so recovery isn't stuck on
  one stale `.bak`, and unknown per-server fields survive a re-save by an older binary.
- **Import naming** — a pasted `npx`/`uvx` package-runner server is named after its
  package (`automem`) instead of `npx`, so runner servers stop colliding on one id.
